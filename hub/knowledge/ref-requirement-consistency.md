# 需求一致性分析规范

所有 PolarCopilot 管理的 Agent（包括单 IDE Agent、集群 Worker、Proxy、Controller）
在向**人类用户**输出最终报告时，必须包含"需求一致性分析"段落。

> 内部管理报告（如 Hub checkpoint、progress_report、worker_activity）不需要包含。
> 只有**直接面向人类用户**的输出才需要：阶段完成报告、任务完成汇报、最后一棒 HANDOFF.md 等。

---

## 1. 方案变更披露义务

**核心原则：AI 可以自主优化方案，但所有变更必须对用户透明。**

- Agent 在执行中发现更好的技术路线 → ✅ 允许自动更改
- 更改后不告知用户 → ❌ 严重违规
- 更改后在给用户的报告中清晰说明 → ✅ 正确做法

即使 Agent 100% 确信自己的方案更优，也必须在报告中体现这一偏离。
用户有权知道"我说了 A，你做了 B"，以及为什么做了 B。

---

## 2. 需求一致性分析格式

以下段落必须出现在每次给人类用户的**最终报告/完成汇报**中：

```markdown
## 需求一致性分析

### 原需求
> [用户原文，逐字引用或精确摘要]

### 原技术路线
- 用户指定：[用户原文中提到的具体技术/方案]
- AI 理解：[Agent 对用户方案的理解和补充]

### 实际使用的技术路线
- [实际采用的技术/架构/方案]
- [如果有偏离] 偏离原因：[为什么改了]

### 实际实现 ← 技术路线反推
基于实际使用的技术路线，项目最终实现了：
- [功能点 1]
- [功能点 2]
- ...

### 一致性判定
- [ ] ✅ 实际实现完全覆盖原需求
- [ ] ⚠️ 实际实现覆盖原需求，且 AI 做了合理改进（列出改进点）
- [ ] ❌ 实际实现偏离原需求（列出偏离点和原因）
```

---

## 3. 批判性自审流程

Agent 写完"需求一致性分析"后，必须进行**批判性自审**。自审的本质不是"解释你做了什么"，而是**模拟用户实际使用，验证流程能不能跑通**。

> ⚠️ **警告**：自审最常见的失败模式是"解释性自审"——Agent 把任何偏离都包装成"合理改进"，让报告看起来总是 ✅ 或 ⚠️。这是毫无价值的。自审的目标是**找到问题**，不是证明自己没错。

### 3.1 心智走查（Mental Walkthrough）

**这是最重要的自审步骤。** 在脑子里模拟用户提出的使用场景，从头到尾走一遍：

1. **还原用户场景**：用户说"我想做 X"，想象用户坐在电脑前，要操作哪些步骤？
2. **逐步模拟执行**：按你实际实现的代码，走每一步：
   - 用户触发操作 → 代码做了什么？返回了什么？
   - 返回的结果用户看得到吗？看到了能理解吗？
   - 下一步操作依赖上一步的什么输出？这个依赖满足了吗？
3. **边界场景**：至少想出 2 个可能出错的场景（如网络断开、数据为空、并发操作），走一遍
4. **最终验证**：用户原来说的那个目标，走完全部流程后，**真的达成了吗？**

```
示例心智走查：
用户需求：给 SOTAgent 加端口申请功能

模拟步骤：
1. Agent 要启动一个服务 → 需要端口 → 写 port_request 到 inbox
   ✓ inbox 路径正确？消息格式对了？
2. SOTAgent 哨兵 30s 后轮询到消息 → 处理 → 查询 port_registry → 分配端口
   ✓ 如果 preferred_port 被占了呢？→ 有回退到 port_range 的逻辑 ✓
3. SOTAgent 写回复到 outbox → Agent 读取 → 拿到 allocated_port
   ✓ 如果 Agent 在 30s 内就需要端口呢？→ 异步等待最多 60s... 🟡 有点慢但可接受
4. Agent 用 allocated_port 启动服务
   ✓ 启动成功 → 注册表正确 ✓
   ✗ 如果这 30s 内另一个进程抢占了这个端口呢？→ ❌ 没有处理！

结论：核心流程能跑通，但有竞态条件风险（分配到启动之间端口被抢）
判定：⚠️ 有改进空间，而非 ✅ 完全覆盖
```

### 3.2 反面论证（Devil's Advocate）

写完心智走查后，**主动站在反对者角度**反驳自己：

- "用户说了 A，我做了 B" → 我的 B 真的比 A 好吗？还是我只是不想做 A？
- "这个改进是合理的" → 用户如果看到我改了，会同意吗？还是会说"我就要 A"？
- "功能已实现" → 如果用户现在打开电脑试用，能成功吗？还是会卡在某一步？

### 3.3 判定标准（从严）

| 判定 | 标准 | 常见错误 |
|------|------|---------|
| ✅ 完全一致 | 用户要 A 做了 A，心智走查全流程畅通 | 把"做了类似的东西"判为完全一致 |
| ⚠️ 合理改进 | 有偏离但能**具体证明**新方案更优（不是"我觉得"） | 把"我没做的功能"包装成"不需要做" |
| ❌ 有偏离 | 心智走查发现流程跑不通，或核心功能缺失 | 自审从不给出 ❌，因为 Agent 不愿承认错误 |

**自审的合格标准**：至少找出一个问题或风险点。如果自审结论是"完美无缺"，大概率是自审没做好。

### 3.4 两种合法的通过情况

**情况 A：实现 = 原需求**
原需求合理，AI 按原需求实现，技术路线一致。心智走查全流程畅通。

**情况 B：实现 ≠ 原需求，但 AI 改进更优**
必须同时满足以下**全部**条件：
- 能具体说明原方案的**具体问题**（不是"不够优雅"这种模糊说法）
- 能具体说明新方案的**可量化优势**（性能提升 X%、代码量减少 Y 行、覆盖了 Z 个边界场景）
- 新方案覆盖了原需求的**核心意图**（目标没变，方式变了）
- **用户如果看到这个改动，大概率会同意**（不确定就问）

## 4. 适用场景

| 场景 | 是否需要 | 说明 |
|------|---------|------|
| 阶段完成报告（phase_complete → 用户） | ✅ 必须 | 每个阶段都要 |
| 最后一棒 HANDOFF.md | ✅ 必须 | 整个任务的总一致性分析 |
| 单 IDE Agent 完成报告 | ✅ 必须 | 直接面向用户的输出 |
| worker → controller 的 result_summary | ❌ 不需要 | 内部报告 |
| controller → proxy 的 progress_report | ❌ 不需要 | 内部报告 |
| Hub checkpoint | ❌ 不需要 | 系统状态存储 |

## 5. 示例

### 好的需求一致性分析

```markdown
## 需求一致性分析

### 原需求
> "给 SOTAgent 加一个端口申请功能，任何服务用端口都要先申请"

### 原技术路线
- 用户指定：端口申请功能
- AI 理解：需要一个端口注册表 + 申请/释放 API + 冲突检测机制

### 实际使用的技术路线
- SQLite port_registry 表存储端口分配
- inbox 消息协议新增 port_request / port_release 类型
- lsof 检测端口实际占用（避免注册表与现实不符）
- **偏离**：用户未提到 lsof 验证，但 AI 认为注册表可能与实际不同步，
  增加了系统级端口检测作为双重保障

### 实际实现 ← 技术路线反推
- ✅ 端口申请：Agent 写 port_request 到 inbox → SOTAgent 分配端口
- ✅ 端口释放：Agent 写 port_release → SOTAgent 释放
- ✅ 冲突检测：preferred_port 被占时自动分配下一个可用端口
- ✅ [额外] 系统级验证：用 lsof 检查端口真实占用状态

### 一致性判定
- [x] ⚠️ 实际实现覆盖原需求，且 AI 做了合理改进
  - 改进点：增加 lsof 系统级验证，防止注册表与实际端口状态脱节
```

### 坏的需求一致性分析（缺失偏离说明）

```markdown
## 需求一致性分析

### 原需求
> "用 fswatch 监听 inbox"

### 实际使用的技术路线
- 用了轮询模式而不是 fswatch（因为 fswatch 在 launchd 下有限制）

### 一致性判定
- [x] ✅ 实际实现完全覆盖原需求
```

**问题**：用户说了 fswatch，实际用了轮询，但判定为"完全覆盖"。
应该判定为 ⚠️ 并说明为什么 fswatch 不行、轮询是更好的选择。
