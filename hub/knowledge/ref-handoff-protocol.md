# 致继任者同步协议

所有 PolarCopilot 管理的 Agent（solo 模式、proxy 模式、worker 完成汇报时）
在每次实质性工作完成后，**必须同步更新致继任者文件夹**。

> 致继任者文件夹是跨 session 的持久化知识传递通道。
> 当前 Agent 的上下文在对话结束后会丢失，致继任者文件夹是唯一可靠的交接手段。

---

## 1. 致继任者文件夹结构

每个项目的 `致继任者/` 文件夹应包含以下文件：

| 文件 | 用途 | 必须 |
|------|------|------|
| `接手文档.md` | Agent 接手导引：去哪报到、读什么文件、怎么用基础设施 | ✅ |
| `需求对齐.md` | 用户需求 ↔ 实际实现的一致性分析（格式见 ref-requirement-consistency.md） | ✅ |
| `相关项目.md` | 能用到的相关项目介绍、路径、技术栈、协作方式 | ✅ |
| `总要求.md` | 用户对整个项目体系的总要求（技术栈、Git 规范、交互规范等） | ✅ |
| `特殊要求.md` | 针对特定子系统/功能的特殊要求 | ✅ |
| `进度.md` | 当前阶段进展、已完成/进行中/待做的具体任务 | ✅ |
| `生态系统分析.md` 等 | 面向人的分析报告（架构、审计等），按需 | 按需 |
| 其他参考文档 | 如 `ref-credential-management.md` 等特定领域规范 | 按需 |

---

## 2. 接手文档.md 的内容要求

接手文档是新 Agent 醒来后读的**第一个文件**，必须包含：

### 2.1 环境定位
- 工作空间路径
- 本项目路径
- 致继任者文件夹路径

### 2.2 身份与职责
- Agent 的角色定义
- 主要职责范围

### 2.3 SOTAgent 报到流程
```bash
cd ~/Polarisor/SOTAgent
./node_modules/.bin/tsx src/cli.ts agent-register <项目路径> <角色>
```

### 2.4 必读文件清单（按顺序）
列出致继任者文件夹中的所有文件，说明阅读顺序

### 2.5 基础设施导引
- SOTAgent 怎么用（常用命令、服务管理、资源调度）
- 项目字典怎么用（跨项目感知，`项目字典/_index.json`）
- PolarCopilot 怎么用（多 Agent 协调）
- 其他项目怎么协作

### 2.6 用户偏好速查
- 语言、交互方式、代码风格、Git 规范等

### 2.7 当前状态概要
- 上一个 Agent 做到哪了
- 下一步该做什么

---

## 3. 需求对齐.md 的内容要求

格式严格遵循 `ref-requirement-consistency.md` 中的规范：

```markdown
## 需求对齐 — [阶段/任务名称]

### 原需求
> [用户原文]

### 原技术路线
- 用户指定：...
- AI 理解：...

### 实际使用的技术路线
- ...
- [偏离原因]

### 实际实现 ← 技术路线反推
- 功能点 1
- 功能点 2

### 一致性判定
- ✅ / ⚠️ / ❌
```

每完成一个阶段/任务，追加一个新的分析段落。

---

## 4. 何时同步

### 4.1 触发条件（必须同步）
- 完成一个阶段/任务的实质性工作后
- 发现并修复了 bug 后
- 用户提出了新需求或修改了需求后
- 对话即将结束前（最后一次同步）

### 4.2 同步内容
| 事件 | 更新的文件 |
|------|-----------|
| 完成一个阶段 | 进度.md + 需求对齐.md |
| 修复 bug | 进度.md |
| 用户新需求 | 总要求.md 或 特殊要求.md + 进度.md |
| 基础设施变更 | 接手文档.md + 相关项目.md |
| 创建/更新分析报告 | 致继任者/ 下对应的 .md 报告 |
| 对话结束前 | 所有有变动的文件 |

### 4.3 同步流程
1. 更新相关文件
2. `git add 致继任者/`
3. `git commit -m "docs: 同步致继任者 — [简述变更]"`
4. `git push origin HEAD`

---

## 5. 与 PolarCopilot SKILL 的集成

### 5.1 solo 模式
在 gsd2-ide-solo SKILL.md 的执行流程中，增加致继任者同步步骤：
- 启动时：读取致继任者文件夹，恢复上下文
- 每完成一个子任务：同步进度.md
- 任务全部完成时：同步所有变动文件 + 需求对齐.md

### 5.2 proxy 模式
Proxy Agent 负责在以下时机同步：
- 分配任务前：确保致继任者信息是最新的
- 收到 Worker 完成报告后：更新进度.md
- 阶段完成后：更新需求对齐.md

### 5.3 CLI solo 模式
同 IDE solo 模式，在 gsd2-cli-solo SKILL.md 中增加同步步骤。

---

## 6. 文件命名规范

- 使用**中文文件名**（与文件夹名 `致继任者` 保持一致）
- 后缀统一 `.md`
- 参考文档以 `ref-` 前缀（如 `ref-credential-management.md`）
